# LangGraph 기반 멀티 에이전트 챗봇 만들기

LangGraph 기반으로 Supervisor가 쿼리를 분석해 적절한 에이전트로 라우팅하는 멀티 에이전트 챗봇 시스템이다.

## 기술 스택

| 레이어 | 기술 | 비전 |
|--------|------|------|
| **프론트엔드** | Next.js + TypeScript | 16.x |
| **백엔드** | FastAPI + Python | 3.12 |
| **AI 오케스트레이션** | LangGraph + LangChain | 0.2.x |
| **LLM** | OpenRouter / Gemini | - |
| **Vector DB** | Pinecone | - |
| **임베딩** | multilingual-e5-large | - |
| **세션 메모리** | Upstash Redis | TTL: 1시간 |
| **인증 + 장기 메모리** | Supabase | - |
| **배포** | Render + Vercel | $0/월 |

## 에이전트 시스템

| 에이전트 | 역할 | 주요 기능 |
|----------|------|----------|
| **Supervisor** | 라우팅 | 사용자 쿼리 분석 → 적절한 에이전트 선택 |
| **ChatAgent** | 일반 대화 | 메모리 명령, 사용자 프로파일링 |
| **CodeAgent** | 코드 | 코드 생성/설명/디버깅/실행 |
| **RAGAgent** | 문서 Q&A | 문서 검색 + 컨텍스트 기반 답변 |
| **WebSearchAgent** | 웹 검색 | Tavily API로 실시간 검색 |
| **ReportAgent** | 보고서 | 다중 소스 연구 결과 종합 보고서 |

## 개발하면서 겪은 문제들

### 1. asyncio.to_thread 사용

동기 SDK 호출이 이벤트 루프를 차단할 수 있다. Pinecone SDK 같은 동기 호출은 `asyncio.to_thread`로 감싸야 한다.

```python
await asyncio.to_thread(client.inference.embed, ...)
```

### 2. Protocol 반환 타입 일치

구현체의 반환 타입을 변경하면 Protocol도 함께 수정해야 한다.

### 3. LLM 토큰화 이슈

일부 모델이 문장 끝 punctuation 뒤 공백 없이 토큰을 생성하는 문제가 있었다. 후처리 함수로 해결했다.

### 4. LLM 프로바이더 선택과 Rate Limit

처음엔 OpenRouter를 사용했는데, 크레딧이 -가 되면서 기본 무료 제공량도 받을 수 없게 됐다. 원인은 정확히 알 수 없었다.

Gemini API로 이동했는데, 여기도 일일 사용량 제한이 빡빡했다. 특히 분당 5회 호출 제한이 문제였다. 현재 구조상 Supervisor가 먼저 LLM을 호출해 쿼리를 분석하고, 그 다음 실제 에이전트가 또 LLM을 호출한다. 한 번의 사용자 요청에 분당 제한을 모두 소진할 수 있는 구조다.

**교훈**: LLM은 최대한 적게 호출해야 한다. Supervisor에서 간결한 방법(키워드 매칭, 작은 모델 등)으로 라우팅하고, 실제 LLM 호출은 최종 응답 생성 시에만 하는 게 좋다.

### 5. 관측 가능성 (Observability)

에이전트 시스템은 디버깅이 어렵다. 어디서 어떤 에이전트가 실행됐는지, 얼마나 걸렸는지 확인이 필요했다.

- **대시보드**: Supabase에 메트릭을 저장해 프론트엔드에서 시각화
- **LangSmith**: LangChain/LangGraph 전용 트레이싱 도구로, 에이전트 실행 흐름을 상세히 추적 가능

## 총평

이 프로젝트를 통해 많은 것을 배웠다.

**좋았던 점**:
- 다양한 무료 서비스(Pinecone, Supabase, Upstash, Render, Vercel)를 직접 사용해볼 수 있었다
- LangGraph로 멀티 에이전트 시스템을 직접 구현해보며 에이전트 오케스트레이션을 이해할 수 있었다
- RAG, 메모리 시스템, SSE 스트리밍 등 챗봇에 필요한 핵심 기능들을 모두 경험해봤다
- 대시보드와 LangSmith로 관측 가능성을 확보해 서비스 상태를 모니터링할 수 있었다

**아쉬웠던 점**:
- LLM 호출이 너무 많다. Supervisor → Agent 구조라서 단순 질문에도 2번 이상 호출된다
- 무료 티어 Rate Limit 때문에 실제 서비스로 쓰기엔 한계가 있다
- LangGraph의 복잡도가 있어, 간단한 챗봇에는 오버엔지니어링일 수 있다

**결론**: 멀티 에이전트 시스템을 학습하고 프로토타이핑하기엔 좋은 프로젝트였다. 다만 실제 서비스로 가려면 LLM 호출 최적화와 비용 문제를 해결해야 할 것 같다.

## FAQ

- **LangGraph가 뭔가요?**
  - LangChain에서 개발한 멀티 에이전트 오케스트레이션 프레임워크입니다. 상태 머신(StateGraph) 기반으로 에이전트 간의 복잡한 워크플로우를 구현할 수 있습니다.

- **Supervisor 패턴이 뭔가요?**
  - 중앙 라우터가 사용자 쿼리를 분석하여 적절한 전문 에이전트로 작업을 위임하는 패턴입니다.

- **RAG는 무엇인가요?**
  - Retrieval-Augmented Generation의 약자로, 문서를 검색(Retrieval)하여 컨텍스트를 구성한 후 답변(Generation)을 생성하는 방식입니다.

- **무료 배포의 한계는?**
  - Render 무료 티어는 15분간 요청이 없으면 서버가 슬립모드합니다. 첫 요청 시 10-30초가 소요됩니다.
